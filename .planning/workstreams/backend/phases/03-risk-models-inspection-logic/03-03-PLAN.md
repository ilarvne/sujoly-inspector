## Phase Goal

**As an** engineer, **I want to** view risk assessments with full factor breakdown and override system-recommended intervals when needed, **so that** I can apply professional judgment while maintaining an audit trail of both system and human decisions.

<objective>
Implement risk assessment persistence (D-04), engineer override with provenance audit trail (D-13), Celery-based risk recomputation (D-05), and a manual recompute API endpoint (D-05 trigger 4). This plan connects the pure Python risk engine (Plan 01) to the database and API, making risk assessments queryable, overridable, and manually recomputable.

Purpose: RISK-06 requires engineer override endpoints with logged provenance. D-04 requires persisted risk assessments with history. D-05 requires event-driven + scheduled recomputation plus a manual API trigger. This plan delivers the complete risk assessment lifecycle: compute → persist → query → override → recompute (manual + scheduled).

**D-05 trigger coverage:**
- Trigger 1 (after new inspection): Implemented in Plan 03-05 via `recompute_structure_risk.delay()` dispatch in `inspection_service.create_inspection`.
- Trigger 2 (after structure update): Requires modifying `structure_service.py` from Phase 2 to dispatch `recompute_structure_risk.delay()` after successful structure update. This is a single-line addition deferred to execution time — the executor adds `recompute_structure_risk.delay(str(structure.id))` after the commit in `structure_service.update_structure`. The Celery task already exists from this plan.
- Trigger 3 (Celery Beat daily): Implemented in this plan via `recompute_all_risks` task + Beat schedule.
- Trigger 4 (manual API trigger): Implemented in this plan via `POST /api/v1/structures/{id}/recompute` endpoint.

**Cross-wave dependency note:** `risk_service.recompute_risk_for_structure` loads `InspectionModel` from Plan 03-05 (Wave 3) to include inspection data in risk computation. Since this plan is Wave 2, the import uses a try/except guard: if `InspectionModel` is not yet available, risk computation proceeds without inspection data (staleness modifier defaults to "never inspected" = 1.5). Full risk accuracy requires all waves complete. This intentional degradation is by design — the try/except guard ensures the system is functional at every wave boundary.

Output: RiskAssessmentModel, risk_service with CRUD + override + recompute + manual recompute endpoint, risk API endpoints (GET risk, POST override, POST recompute), Celery tasks for recomputation.
</objective>
