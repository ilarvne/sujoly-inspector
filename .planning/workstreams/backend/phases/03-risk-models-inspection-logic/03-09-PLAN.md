    **Gaps closed:** BLOCKER — "PDF export broken in Docker runtime" (#3); WARNING — "GeoJSON raw geometry" (#8), "CSV lang-unaware name" (#9), "MinIO connection leak" (#10)

    **A. Dockerfile — copy templates to runtime image:**

    In the runtime stage (after `FROM python:3.12-slim-bookworm`), add a COPY line between the existing COPY lines (after line 35 `COPY --from=builder /app/src /app/src`):
    - `COPY --from=builder /app/templates /app/templates`

    This ensures the Jinja2 templates directory exists in the runtime image so `export_service.py` can resolve `/app/templates/inspection_report_{lang}.html`.

    **B. export_service.py — fix GeoJSON geometry serialization:**

    In `export_structures_geojson` (around line 337-346), the `geometry` attribute from the ORM model is a raw WKB/PostGIS object that cannot be JSON-serialized. Fix by converting to a GeoJSON-compatible format:

    Replace the direct `getattr(struct, "geometry", None)` with a call that serializes the geometry properly. Use `json.loads()` on the GeoJSON output from GeoAlchemy2's `ST_AsGeoJSON` via a helper, or check if the structures route's GeoJSON format already handles this correctly and reuse that pattern.

    The simplest approach: import `json` and `from geoalchemy2.shape import to_shape`, then:
    - `geometry = to_shape(getattr(struct, "geometry", None)) if getattr(struct, "geometry", None) is not None else None`
    - `geometry_geojson = json.loads(geometry.geojson) if geometry else None`

    This converts the WKB geometry to a Shapely geometry, then to GeoJSON dict.

    **C. export_service.py — fix CSV name column to be lang-aware:**

    In `export_structures_csv` (around line 251), the first CSV column always uses `name_ru`:
    - Replace `getattr(struct, "name_ru", "")` with `getattr(struct, f"name_{lang}", "")` so Kazakh export uses `name_kk` and English uses `name_en`, with fallback to `name_ru` if the lang-specific name is None.

    **D. export_service.py — fix MinIO connection leak:**

    In `export_inspection_report_pdf` (around line 397-416), `minio_service.client.get_object()` returns a response object with an HTTP connection. The current code calls `response.read()`, `response.close()`, `response.release_conn()` separately, but on exception the connection leaks.

    Wrap in a try/finally block that always releases the connection, or use a context manager pattern. The MinIO Python SDK's `get_object` response object supports the context manager protocol:
    - Replace the bare `response = minio_service.client.get_object(...)` + manual close/release with:
      `with minio_service.client.get_object(photo.minio_bucket, photo.minio_object_key) as response:`
      `    photo_bytes = response.read()`

    This ensures the HTTP connection is released even if `response.read()` raises an exception.
